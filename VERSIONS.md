- **0.0.41** ‚Äî Tailwind v4 migration. The Dependabot-merged `tailwindcss` 3.4.19 ‚Üí 4.3.0 bump (commit `6e9fe19`) silently broke `npm run build` because v4 moved its PostCSS integration into a separate `@tailwindcss/postcss` package and changed config loading. - **0.0.39** ‚Äî Security #8: ICS parsing DoS hardening. The pure parser is extracted out of `src/lib/ics.ts` into a worker-safe module `src/lib/icsParse.ts` (no Supabase imports, no DOM access) so it can be loaded from a Web Worker without bundling the auth client. A global per-feed cap `MAX_INSTANCES_PER_FEED = 5000` is enforced inside `parseIcs`: every push goes through a small helper that throws an `IcsTooLargeError` once the cap trips, so a feed expanding to 10k+ instances (e.g. a hostile daily-RRULE-since-2010) aborts cleanly instead of pinning the main thread. New worker entry `src/workers/icsParser.worker.ts` re-exports a `ParseResponse` discriminated union (`{ ok: true, events } | { ok: false, error }`) and runs the parse on a message handler. `src/lib/ics.ts` adds `parseIcsInWorker(text, calendarId)` which spawns the worker via Vite's `?worker` import, posts the text, and resolves the events ‚Äî wrapped in a 5s `setTimeout` reject that terminates the worker on timeout (also fires on any pathological case the cap somehow misses). `fetchIcsCalendar` now calls `parseIcsInWorker` instead of synchronous `parseIcs`. Existing callers that imported `parseIcs` / `IcsEvent` from `./ics` keep working via re-export. Combined with the existing 5 MiB body cap in the `fetch-ics` Edge Function, the entire pipeline now has bounded work at every step.

No new destructive flow exists in-app; Phase B (nightly server-side snapshots) and Phase C (soft delete) remain deferred per `SECURITYPLAN.md`.
- **0.0.37** ‚Äî Security #3: harden the `fetch-ics` Edge Function against SSRF. The single-file handler at `supabase/functions/fetch-ics/index.ts` is split into a thin request handler and a pure helper module `safe-fetch.ts` (with a sibling `safe-fetch_test.ts` of `Deno.test` cases for the URL / IP allow-list helpers). The new `safeFetch(rawUrl)` enforces: http(s) scheme only, ports 80/443 only, no embedded credentials; DNS pre-resolution (A + AAAA) with every returned address checked against private/reserved ranges (RFC1918, loopback, link-local incl. `169.254.169.254` cloud metadata, CGNAT, multicast, IETF reserved; IPv6 `::1` / `::` / `fc00::/7` / `fe80::/10` / `ff00::/8`; IPv4-mapped `::ffff:0:0/96` re-checked as v4); manual redirects with a 5-hop cap, each `Location` re-run through the full allow-list; 10s `AbortController` timeout (returns HTTP 504); 5 MiB streamed body cap (mid-stream abort); `Content-Type` allow-list of `text/calendar` / `text/plain` / `application/octet-stream`; and a `BEGIN:VCALENDAR` sniff on the first bytes so 200-OK HTML error pages don't slip through. Response shape is unchanged for the client (`{ text }` on success, `{ error }` with a clean status on failure) ‚Äî `src/lib/ics.ts` needs no changes. Deploy with `supabase functions deploy fetch-ics`.
- **0.0.36** ‚Äî Security #5: FK ownership checks in RLS. Existing INSERT/UPDATE policies on `tasks`, `custom_list_items`, and `task_templates` only checked `auth.uid() = user_id` on the row being written ‚Äî a signed-in user could supply someone else's `project_id` / `list_id` / `template_id` in the payload. New migration `supabase/migrations/0009_fk_ownership_rls.sql` drops and recreates those six policies (insert + update on each of the three tables) so the `with check` clause additionally requires `exists (select 1 from <parent> where id = <fk> and user_id = auth.uid())` for every FK to an owned table: `tasks.project_id` and `tasks.template_id` (both nullable), `custom_list_items.list_id` (not null, unconditional), and `task_templates.project_id` (nullable). Single-owner deployments don't notice, but the gap was a horizontal-privilege issue waiting for the second user.
- **0.0.35** ‚Äî Security #6: explicit Data API grants for `task_templates` + cross-table RLS assertion.
