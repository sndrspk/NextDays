# Versions

A running log of changes. Newest at the top. One line per change.

- **0.0.34** — Security #4: Dependabot + npm audit in CI. New `.github/dependabot.yml` opens weekly grouped (minor+patch in one PR) npm update PRs and monthly github-actions update PRs, both labelled `dependencies`. New `.github/workflows/audit.yml` runs `npm audit --audit-level=high` on every pull_request against main, every push to main, and on a weekly Monday cron (so advisories filed against the existing lockfile still surface). Current baseline is clean (`npm audit --audit-level=high` reports `found 0 vulnerabilities`), so the step is blocking from day one rather than `continue-on-error`.
- **0.0.31** — Show / hide completed tasks toggle on the Calendar and Focus screens. New shared `CompletedToggle` (`src/components/common/CompletedToggle.tsx`) renders as a two-button segmented control: the left side shows a stricken-through "T" SVG icon (visible state); the right side is intentionally empty (hidden state). On **Calendar**, the toggle sits in the header row next to the existing day-count toggle and defaults to **visible**; the filter is applied inside `DayColumn` before sorting via `tasks.filter((t) => !t.completed)`. On **Focus**, the toggle sits in the page header next to the title and defaults to **hidden**; partitioning into Overdue / Due today / Scheduled for today now happens against the filtered list, and each section uses a new `partitionAndSort` helper so completed rows sink to the bottom when shown. `useFocusTasks` no longer applies the server-side `completed = false` filter — completed tasks are now fetched alongside active ones so the view can flip the toggle without a refetch. State lives in local `useState` in each view (not persisted); defaults return on every page refresh as specified.
- **0.0.30** — Focus view: move "Calendar events today" section to the top of the page, directly below the "Focus" header and above the quick-add input and all task sections (Overdue / Due today / Scheduled for today).
- **0.0.29** — Move the ICS fetch server-side. The original client-side `fetch(url)` failed against virtually every public ICS endpoint because providers don't send `Access-Control-Allow-Origin`. New Supabase Edge Function `supabase/functions/fetch-ics/index.ts` accepts `{ url }`, validates it's public http(s), proxies the GET, and returns `{ text }` (or `{ error }` on failure). JWT verification stays on so only authenticated users of the project can use the proxy. `fetchIcsCalendar` in `src/lib/ics.ts` switches from a raw `fetch` to `supabase.functions.invoke("fetch-ics", { body: { url } })` — the user's JWT is attached automatically. The Settings panel copy drops the CORS caveat and now reads "Calendars are fetched via the fetch-ics Supabase Edge Function, so any ICS host works." A missing deploy surfaces as a clear "Edge Function isn't deployed yet" error. Deploy with `supabase functions deploy fetch-ics`.
- **0.0.28** — ICS calendar subscriptions are now per-account instead of per-device. New migration `supabase/migrations/0006_ics_calendars.sql` adds an `ics_calendars` table (`id uuid pk default gen_random_uuid()`, `user_id uuid default auth.uid()` with `on delete cascade`, `url`, `name`, `colour`, `created_at`) with RLS enabled, per-command policies scoped to `auth.uid() = user_id`, and explicit GRANTs to anon/authenticated/service_role (matching the Data API policy from migration 0005). `IcsCalendarRow` joins `Project` & friends in `types/index.ts`. New `src/hooks/useIcsCalendars.ts` exposes a `useQuery` ordered by `created_at` plus create / update / delete mutations; delete additionally clears the local `nextdays:icsCalendarCache:<id>` entry. `useExternalEvents` reads the list from the hook instead of `SettingsProvider`; `IcsCalendarsSection` is rewired to the mutations (pending states disable inputs, errors surface inline). `SettingsProvider` is back to font / dayCount / fontSize only — all ICS pieces removed, including the `nextdays:icsCalendars` localStorage entry. `CalendarStrip` and `FocusView` both call `useIcsCalendars()` directly to colour the event cards. **Backup now includes `ics_calendars`** since they're an account-level resource: `exportAll()` selects the table, `wipeAll()` truncates it in replace mode, and `importBackup()` upserts the rows. The envelope keeps `schema_version: 1` and treats `ics_calendars` as **optional** so v1 backups taken before this table existed still import. `BackupCounts` gains an `ics_calendars` field and the post-restore summary mentions calendar count when non-zero. The parsed-events cache stays per-device in `localStorage` because it's purely network output. Settings panel copy updated from "Stored locally on this device" to "Synced across every device signed in to this account".
- **0.0.27** — Wishlist item 6, Path A: read-only `.ics` calendar overlay (client-side fetch + cache). New `src/lib/ics.ts` parses with `ical.js`, expands RRULEs across `today - 1d → today + 60d`, and caches results in `localStorage` under `nextdays:icsCalendarCache:<id>` with a 15-minute TTL. `IcsCalendar = { id, url, name, colour }` list lives in `SettingsProvider` (`nextdays:icsCalendars`) — settings-only, intentionally **not** part of the backup envelope. `useExternalEvents` runs one TanStack query per calendar (cached entries hydrate `initialData` so the UI paints instantly) and returns `{ events, byDate, errors, fetchedAt, isFetching, refresh }`. New `IcsCalendarsSection` slots into Settings between Tags and Backup & Restore: URL + name + colour-swatch form (reuses `PROJECT_COLOURS`), per-row inline rename, colour-change popover, last-synced timestamp / inline error, remove with confirm, and a "Refresh all" button. New `EventCard` renders a translucent tinted card (`<colour>22` background + 3px coloured left border) above the task list in `DayColumn`; timed events get a `HH:MM` monospace prefix in the calendar colour, all-day events show title only. Events sit before tasks in every day column. `FocusView` gains an "Calendar events today" section between "Due today" and "Scheduled for today" — hidden when empty, counted into `total` so the Inbox-zero state still fires correctly. Events are non-interactive (no click target). CORS caveat is called out in the Settings panel because most providers besides Google block cross-origin GETs on their public ICS endpoints.
- **0.0.26** — Wishlist item 4: drag-and-drop tasks between calendar day columns. `@dnd-kit/core` + `@dnd-kit/utilities` added as dependencies. `CalendarStrip` is now a `DndContext` with `MouseSensor` (6px activation distance), `TouchSensor` (200ms delay), and `KeyboardSensor`. Each `TaskCard` is a draggable (id = task uuid, carries the full `task` object in `data`). Each `DayColumn` is a droppable with id `day:YYYY-MM-DD` and shows an `accent-100` tint when a card is held over it. On drop, `useMoveTask` (`src/hooks/useMoveTask.ts`) optimistically rewrites the task's `scheduled_date` in the TanStack Query cache, fires a Supabase `UPDATE`, and rolls back if the server errors. If a task's `start_date` would hide it at the new date (`start_date > target`), it is pulled forward to the target automatically. A `DragOverlay` renders a lifted, rotated ghost card while dragging. Same-column drops are a no-op.
- **0.0.25** — Due-soon / overdue indicator in `TaskCard` changed from the 🔔 emoji to a plain bold `!` for a cleaner look that fits the app's minimal style.
- **0.0.24** — Three changes in one drop. (1) **Inline `@Project` / `#tag` quick-add syntax.** Every quick-add input — `QuickAdd` in `DayColumn`, `FocusQuickAdd` in `FocusView`, `ProjectQuickAdd` in `ProjectView` — now pipes the typed title through `parseTaskTitle()` (new `src/lib/parseTaskTitle.ts`) before saving. `@word` is matched case-insensitively against existing project names (first match wins; unknown `@`s stay in the title); `#word` becomes a tag (deduped case-insensitively; tags don't have a table of their own, so "creating" one means including it in the new row's `tags` array). Matched tokens are stripped and surrounding whitespace is collapsed. `useCreateTask` gained an optional `tags` field. In `ProjectQuickAdd` an explicit `@Project` overrides the open project; without one the current project is still assigned. (2) **Tags panel in Settings.** New `TagsSection.tsx` between Text size and Backup & Restore lists every distinct tag across `tasks` + `task_templates` (aggregated by `useTags`) with its usage count, an inline click-to-edit rename (Enter/blur commits, Escape cancels), and a Delete button with a `window.confirm` naming the tag and the number of affected items. Rename + delete are implemented by `useRenameTag` / `useDeleteTag` (`src/hooks/useTags.ts`) — both `.contains("tags", [target])`-query the matching rows and rewrite each `tags` array in a loop; rename de-dupes case-insensitively so `foo → Bar` collapses any pre-existing `Bar`. Mutations invalidate `["tags"]`, `["tasks"]`, and `["task_templates"]` on success. (3) **Sort order in day columns and focus sections.** Active tasks are ordered by (a) earliest `due_date` first (no-due-date sinks to the bottom of the active list), then (b) latest `scheduled_date` first, then `sort_order` as a tiebreaker. Completed tasks still sink to the bottom in completion-time order. The shared comparator `compareActiveTasks` lives in `DayColumn` and is imported by each of `FocusView`'s three sections (Overdue / Due today / Scheduled for today). The old "pinned-when-due-within-one-day" bucket in `DayColumn` is dropped — pinning is now an emergent property of the primary sort.
- **0.0.23** — Settings gains a **Text size** option with three choices: **Normal** (default), **Larger** (+10%), **Largest** (+20%). Implemented as a segmented control in `SettingsView` underneath the Appearance / font panel; preview text inside each button is rendered at the target scale so the difference is visible before committing. State lives in `SettingsProvider` (`fontSize` / `setFontSize`) and persists in `localStorage` under `nextdays:fontSize`. `FONT_SIZE_OPTIONS` exports the three options as `{ id, label, scale }` tuples (1, 1.1, 1.2). Applied app-wide by writing CSS `zoom` (and a matching `--app-font-scale` variable) onto `<html>` whenever the value changes — `zoom` was chosen over a root `font-size` change because the app uses `text-[Npx]` Tailwind arbitrary values that wouldn't respond to rem scaling; `zoom` proportionally scales typography, spacing, and hit targets in one shot. Mobile layouts stay usable because viewport-relative media queries evaluate before zoom is applied, so the calendar still stacks vertically on phones at 1.20×. Existing font-family setting is untouched.

- **0.0.19** — Backup & restore. New `src/lib/backup.ts` exposes `exportAll()` (parallel `SELECT *` on `projects` / `task_templates` / `tasks` / `custom_lists` / `custom_list_items` into one versioned JSON envelope with `schema_version: 1` + `app: "nextdays"` + `exported_at`), `downloadBackup()` (Blob + anchor click → `nextdays-backup-YYYY-MM-DD-HH-MM-SS.json`), `parseBackup()` (JSON shape + app + schema_version validation), and `importBackup(envelope, mode)`. Two import modes: **Merge** uses `upsert({ onConflict: "id", ignoreDuplicates: true })` so existing rows are kept; **Replace** wipes children-then-parents via `delete().neq("id", SENTINEL_UUID)` first (RLS scopes to the current user), then bulk-inserts parents-then-children. `user_id` is stripped from every payload row before insert so the DB default `auth.uid()` fills it from the current session — the same backup works across accounts. New modal lives in `src/components/settings/BackupPanel.tsx` (centered card, Escape/backdrop to close), triggered from a new download icon in the sidebar footer between the user chip and sign-out. Replace mode shows a `window.confirm` with row counts and a "cannot be undone" warning. After a successful import, `qc.invalidateQueries()` refreshes every view.

- **0.0.13** — Recurrence reworked from scheduled-date-anchored (with offsets) to per-field. Migration `0004_recurrence_per_field.sql` wipes existing templates, drops `rrule`/`dtstart`/`start_offset_days`/`due_offset_days`, and adds `start_rrule`/`start_dtstart`/`due_rrule`/`due_dtstart` plus a CHECK requiring at least one pair to be populated. `RecurrenceEditor` now renders two independent sections — one for the start date, one for the due date — each disabled until the matching date is set on the task.
